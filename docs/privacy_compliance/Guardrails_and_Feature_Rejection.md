# Guardrails and Feature Rejection

## Core principle

Guide, do not spy.

## Product guardrails

The product must:

- stay launcher-first
- be honest about Android limitations
- avoid surveillance framing
- avoid secret monitoring
- minimize permissions
- avoid child data collection
- avoid cloud dependency at launch
- remain parent-friendly and child-safe

## Rejected MVP features

Do not build in MVP:

- message monitoring
- call recording
- microphone monitoring
- location tracking
- web history spying
- remote parent dashboard
- cloud account dependency
- AI chat assistant for child
- ads
- hidden data collection
- social feed
- full lockdown claims
- “cannot be bypassed” claims
- enterprise kiosk claims
- device-owner mode

## Conditional future features

These may be considered later only after separate documentation:

- local backup/restore
- scheduled mode switching
- multiple child profiles
- school/kiosk edition
- device-owner managed device mode
- parent cloud dashboard

## Feature evaluation test

Before adding any feature, ask:

1. Does it help the parent simplify the child phone experience?
2. Does it require child data collection?
3. Does it introduce surveillance concerns?
4. Can it work offline?
5. Can we explain it honestly in Play Store wording?
6. Does it overpromise Android control?
7. Does it need a sensitive permission?
8. Is it essential for MVP?

If the feature fails privacy, trust, or honesty checks, reject or defer it.
